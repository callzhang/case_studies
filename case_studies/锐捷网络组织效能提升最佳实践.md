# 

优化需求管理流程，承接需求管理 SOP​

锐捷原本使用自研的 PMS 系统来管理产品的全生命周期，但随着组织发展，现有 PMS 无法满足需求的全生命周期管理：​

  * 各事业部没有统一的需求管理规则，缺乏统一的工具来沉淀 SOP，信息不透明；​



  * 需求收集缺少统一、便捷的提交入口，需求分析缺少拉通机制，需求状态缺少跟踪和同步机制；​



  * 由于线下数据填写工作量大、不及时等现象，线上数据不准确，且原有 PMS 没有强大的度量模块支撑，很难发现需求管理过程中的问题，也无法监控需求处理人员的绩效。​



飞书项目可视化的工作流管理能力、高自由度的配置能力、平台级别的开放能力，解决了锐捷面临的痛点。​

通过飞书项目，锐捷重新搭建了需求管理流程 SOP、定义了各角色的分工和交付物，并通过可视化流程引擎将其承载起来，让信息在组织内部充分流动。同时，锐捷设计了统一的需求提交表单，用户可以在飞书、工作台、移动端、网页等多个渠道提交。需求卡还能够对齐同类客户的需求、竞品的解决方案，打破单一场景、收获全局视角。飞书项目也让锐捷的需求进展透明化，不管是新建还是流转，整体进展都会自动同步到需求相关人。在度量层面，需求的吞吐、处理速率、产品经理的绩效，都清晰可见。​

优化协作流程，需求评审会议效率提升 30%​

锐捷会从企业内外部收集各种产品需求，“高价值且复杂的需求”最终进入需求评审节点。但是需求评审会的过程很难记录下来，会议中提出的修改记录也只能体现在会议纪要中，一次需求评审会需要召开多轮会议才能搞定，效率低下。​

  * 会前：产品经理需要使用 Word、Viso、Axure 多个工具才能完成一篇需求文档，工具互不兼容，有些信息无法在 Word 中展示，损失可读性。Word 文档也不易传播、无法互动、无法随笔记录疑问，阻碍了会前阅读的落地。​



  * 会中：产品经理讲解需求文档时，研发会开单独的文档记录疑问，互动效果不好；讲解后大家依次发言，话题分散不聚焦，也拉低了会议效率；专人做会议记录，但口头发言多、问题讨论发散，整理费力、记录结果易失真。​



  * 会后：锐捷通过邮件同步资料、纪要和录制文件，过程繁琐、时效性较差、视频观看不方便，真正去查阅的人较少。产品经理基于会上修改意见修改需求文档后，还要召开二轮、多轮需求评审会，效率极低。​



借助飞书强大的协作能力和一体化能力，锐捷优化了原有的流程，开起了更加高效的需求评审会。​

使用飞书云文档，能插入流程图、思维导图、原型链接等信息，产品经理无需频繁切换系统，即可完成 PRD 编写工作。研发阅读 PRD 时，也能将疑问和建议直接划词评论沉淀在文档右侧。参会者也能在会中划词评论，将想法全部沉淀在一处；产品经理讲解完毕聚焦这些评论、围绕问题讨论，高效不跑题；同时，评论都会自动留存下来，信息还原更准确。会后，自动生成的飞书妙记同时包含文字和视频，回顾起来更加方便。产品经理在文档中修改完毕后，直接@相关人同步进展，大家基于文档讨论和协作完善PRD，不再需要进行冗长的二次评审。​

实时监控、快速响应，告警信息 100% 闭环处理​

在锐捷，MES 系统作为生产过程的执行终端，负责整体锐捷的生产任务执行。系统运行过程产生了大量的关键数据，存在海量、多样性、及时性的特征。同时，产前的众多数据需要对接不同的系统，每日的调度任务信息多达 2000+，接口处理信息有上千条，不同的预警任务也近 50 条。这给锐捷带来了如下痛点：​

  * 提醒工具单一：消息提醒只能通过系统查询和邮件预警功能实现，方式单一、效率低下​



  * 预警信息通知不及时：问题产生到反馈至IT进度迟滞，无法做到问题及时处理​



  * 缺乏任务过程监控：告警执行结果缺少跟进反馈机制，仍需要人工沟通获知结果​



  * 数据没有汇总分析：无法汇总分析不同系统的错误业务数据对生产的影响，阻碍相应系统优化调整​



通过轻量化的集成，锐捷用飞书多维表格将告警信息汇总到一处，让告警处理过程清晰可见。​

使用 API 接口，锐捷的 MES 系统能够将告警信息自动汇总至多维表格、自动通知相关负责人。点击通知卡片，负责人就可以直接查看告警详情，缩短问题处理路径、提高问题处理效率。而在任务处理过程中，多维表格的自动化流程能够实时监控告警状态变化、自动通知到人，无需重复沟通即可快速了解告警处理进度。最后，仪表盘会自动汇总分析异常数据、关键数据情况，为业务优化指引方向。​
